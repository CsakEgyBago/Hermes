The Q3 server outage was caused by a memory leak in the caching layer. A hotfix was rolled out that added a maximum size limit to the cache. Root cause analysis is still ongoing for a secondary latency spike. Heap snapshots confirmed the growth pattern matched the suspected cache object. The fix has since been backported to the staging environment for regression testing. A ticket was filed to add automated alerts for unbounded memory growth. Pairing with the infrastructure team helped narrow down the affected service within twenty minutes.

Profiling data collected during the incident showed the cache object growing at a roughly linear rate for the first ninety minutes before the growth curve steepened noticeably. A follow-up experiment reproduced the leak in a sandboxed environment by replaying production traffic at half load, which made it much easier to validate the fix without risking another outage. The team also discovered a second, unrelated code path that could theoretically trigger a similar leak under different conditions, and a separate ticket was opened to track that risk. Load testing after the fix showed memory usage staying flat for over six hours under sustained traffic, which the team is treating as a reasonable confidence signal ahead of a wider rollout. Notes from the debugging session have been added to the team's internal runbook so the next engineer who hits something similar has a starting point.
