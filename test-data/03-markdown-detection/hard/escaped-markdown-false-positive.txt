This note tries to escape all of its Markdown so nothing renders: \**escaped bold** and \`escaped code\` should both show up as plain text with the backslashes visible, not as real formatting. The author only escaped the leading character of each pair, which is a common mistake — real Markdown requires escaping every special character involved to fully suppress it. Because a naive heuristic just checks whether the \*\*...\*\* and `...` substrings exist anywhere in the text, it doesn't know or care about the stray backslashes sitting next to them, so both signals still fire and this still gets auto-detected as Markdown despite the author's best effort to escape it.

The author apparently ran into this same issue on a previous note as well, judging by a comment left further down: "tried escaping the asterisks again and it still shows up bold somehow, not sure what I'm doing wrong here." A colleague later explained in a reply that CommonMark-style escaping requires a backslash immediately before every individual special character, not just one at the start of the run, but by then the note had already been sent out several times with the same mistake. This whole file is really a small case study in how easy it is to get manual Markdown escaping subtly wrong, and how a naive substring-based heuristic will happily reproduce that same mistake rather than catching it.
